# DCU BF16×FP32 GEMM (gfx936)

Optimal direct HIP kernel for BF16×FP32→FP32 GEMM on Hygon DCU gfx936 (80 CUs, DTK-26.04). N=256, K=3072 fixed; M=1..4096 variable.

## Build & Run

```bash
# Standalone benchmark (tuned for 86 M-values, 1-4096)
hipcc --offload-arch=gfx936 -O3 kernels/gemm_dispatch.cu -o artifacts/gemm_dispatch && ./artifacts/gemm_dispatch

# rocBLAS reference
hipcc --offload-arch=gfx936 -O3 kernels/bench_rocblas.cu -lrocblas -o artifacts/bench_rocblas && ./artifacts/bench_rocblas

# Precision compare (v_pk_fma vs MMAC vs rocBLAS vs CPU double)
hipcc --offload-arch=gfx936 -O3 kernels/bench_precision_compare.cu -lrocblas -o artifacts/bench_precision_compare && ./artifacts/bench_precision_compare

# Step=64 vs step=32 verification
hipcc --offload-arch=gfx936 -O3 kernels/verify_k64.cu -o artifacts/verify_k64 && ./artifacts/verify_k64

# PyTorch extension
cd warp && python setup.py build_ext --inplace
python test_warp_gemm.py --full --tflops

# ISA disassembly
dccobjdump --inputs=./artifacts/gemm_dispatch --architecture=gfx936 --show-sass --show-resource-usage

# PMC profiling
hipprof --pmc ./artifacts/gemm_dispatch
hipprof --pmc --pmc-type 3 ./artifacts/gemm_dispatch  # CSV output
```

## Precision Paths

| Path | avg_rel | When to use |
|------|---------|-------------|
| **v_pk_fma** (FP32) | ~5e-6 | Strict FP32 precision required |
| **TF32 MMAC** (v_mmac_f32_16x16x8_tf32) | ~2e-3 | Perf-critical, B→TF32 truncation acceptable |

rocBLAS on gfx936 also uses TF32 MMAC (confirmed via hipblasGemmEx + dccobjdump). Its avg_rel appears FP32-like (~5e-6) because most elements don't cancel catastrophically, but max_rel is TF32-level (~1e-3). Catastrophic cancellation on near-zero residual dot products can produce >100% max_rel in both our MMAC and rocBLAS — this is inherent to TF32's 10-bit mantissa.

## Dispatch Architecture

### gemm_dispatch_tf32 (entry point at `kernels/gemm_dispatch.cu:1755`)

9 bands, 32×64+LDS tile (4 WF, 256 threads, step=64):

| M range | BK | Tile |
|---------|----|------|
| ≤32 | 128 (×24 slices) | 16×32 |
| ≤64 | 192 (×16) | 32×64+LDS |
| ≤128 | 256 (×12) | 32×64+LDS |
| ≤224 | 384 (×8) | 32×64+LDS |
| ≤256 | 256 (×12) | 32×64+LDS |
| ≤384 | 384 (×8) | 32×64+LDS |
| ≤512 | 512 (×6) | 32×64+LDS |
| ≤2048 | 768 (×4) | 32×64+LDS |
| >2048 | 1024 (×3) | 32×64+LDS |

3D grid: `blockIdx.z * BK` selects K-range. atomicAdd for C accumulation. `C` must be zeroed before each launch.

### gemm_ABT_64x64_ldsB_dispatch_tf32 (entry point at `kernels/gemm_ABT_dispatch.cu:961`)

Same 9-band structure but 64×64 tile with LDS A+B. APAD/BPAD=34 eliminates all LDS bank conflicts. BK=1024/768/512/384/256/192 bands matching gemm_dispatch. 16×32 tile fallback for M≤32.

## Critical Design Rules

- **Never** convert B to BF16 — neither in memory nor at compute time
- **`__builtin_hcu_mmac_f32_16x16x16_bf16` is broken** on gfx936 (DTK-26.04, 503% error). Use `v_mmac_f32_16x16x8_tf32` instead (TF32, not BF16)
- **K-slice stride bug class**: K-loop stride must equal the number of K elements actually loaded. `t += 32` with only 1 uint32_t load (2 elements) = 25% K coverage. Load 4 uint32_ts at offsets 0,8,16,24 for correct 32-element stride. Step=64 uses stride=36 with two 32-element load+sync+MMAC+sync cycles
- **Step=64 is optimal** (not step=128). Step=128 doubles loop body (910 vs 455 instr) causing I-cache pressure — measures 98-101% of step=64. Benefit of step=64 over step=32 comes from halved loop iterations (16 vs 32 per BK=1024 slice), same sync density (4 syncs/64K)
- **LDS A sharing yes, LDS B sharing no**: A in LDS (+50% perf from eliminating redundant A loads). B in LDS regresses (−25%) because B loads move before `__syncthreads`, preventing compiler from software-pipelining B loads with MMAC
- **APAD/BPAD=34** for bank-conflict-free LDS (stride 34 mod 32 = 2, coprime → each lane hits distinct bank). Float layout moves BF16→float conversion from MMAC phase to load phase
- **All kernel launches must pass `hipStream_t stream`** when called from PyTorch extension. Supplied by `at::cuda::getCurrentCUDAStream()`. Without this, `torch::zeros` memset races with GEMM kernel. Standalone `main()` passes `0`
- **`-DWARP_LIB`** compile flag excludes stand-alone `main()` from dispatch source files when building the PyTorch extension (wraps `main()` in `#ifndef WARP_LIB`)
- **v_pk_fma rules**: row-pairing gives 4 FLOP/instruction (vs 2 for column-pairing). Named-scalar accumulator variables beat arrays (compiler can't optimize arrays). ILP/MLP > occupancy for M≤256. No LDS benefit for M≤256
- **Inline compound literals in MMAC builtins are safe**: `foo({a,b})` in builtin calls compiles to correct SASS. Only `(v2i){a,b}` C99 compound literal in *assignment* context is buggy
- **`#pragma STDC_FP_CONTRACT OFF`** if v_pk_fma precision critical (DCC 26.04 has known FMA contraction precision loss)

## Hipprof PMC Profiling

```bash
hipprof --pmc ./artifacts/gemm_dispatch                    # compact TXT
hipprof --pmc --pmc-type 3 ./artifacts/gemm_dispatch       # CSV with all columns
```

Key TXT columns: kernel time, ALU util %, L1 active/stalled %, L2 hit/conflict %, LDS bank conflict %. hipprof perf (GFLOPS) assumes 2 FLOP/VALU (FP32 FMA) — **undercounts MMAC** where 1 `v_mmac_f32_16x16x8_tf32` = 4096 FLOPs but counts as 1 VALU.

## dccobjdump

```bash
dccobjdump --inputs=./artifacts/gemm_dispatch --architecture=gfx936 --show-sass --show-resource-usage
```

Key per-kernel info: VGPR, SGPR, LDS size, instruction count, barrier count.
